# Research

When conducting research, follow this structured workflow:

## 1. Define the Research Question
- Clarify what specifically needs to be answered
- Identify the scope: broad landscape survey vs. specific technical answer
- Determine the audience for the findings (Jim's own use, partner content, customer-facing)

## 2. Source Strategy
Use sources in this priority order:

| Source | Tool | Best For |
|--------|------|----------|
| Microsoft Learn | `microsoft_docs_search`, `microsoft_docs_fetch` | Official Microsoft product docs, best practices, architecture guidance |
| Microsoft Code Samples | `microsoft_code_sample_search` | Implementation examples, SDK usage, API patterns |
| Web Search | `web_search` | Industry trends, competitive analysis, community perspectives, recent developments |
| OneDrive / Workspace | File search, `ask_work_iq` | Prior work, internal documents, team-shared resources |
| `_shared-resources\references\` | File read | Previously curated reference materials |

## 3. Gather & Organize
- Search multiple sources to get a well-rounded view
- Take structured notes as you go — don't just dump raw results
- Track sources for citation
- Flag conflicting information between sources

## 4. Synthesize Findings
Present research as:

### Quick Answer (if straightforward)
- Direct answer in 2-3 sentences
- Key source with link

### Research Brief (if complex)
1. **Question**: What we set out to answer
2. **Key Findings**: 3-5 bullet points with the most important takeaways
3. **Details**: Deeper analysis organized by sub-topic
4. **Comparison Table**: If evaluating options (features, pros/cons, pricing)
5. **Recommendation**: Jim's best path forward based on findings
6. **Sources**: Numbered list of references with URLs

## 5. Save & Cite
- Always include source URLs for verification
- Save research outputs to the active project folder
- If the research is broadly useful, suggest saving to `_shared-resources\references\`
